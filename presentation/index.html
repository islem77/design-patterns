Section 1

Introduction to Design Patterns

Definition and Relevance

  • Fundamental principles in software development
  • Standardized solutions to common problems.
  • They play a crucial role in improving:
    • code quality
    • maintainability
    • scalability

Categories of Design Patterns

  • Creational Patterns: Focus on object creation mechanisms, providing flexibility in object creation and hiding the creation logic.
  • Structural Patterns: Deal with object composition and class inheritance, emphasizing the ways objects can be composed to form larger structures.
  • Behavioral Patterns: Focus on how objects interact and communicate with each other, defining patterns for communication between objects.
Section 2

Creational Design Patterns

Creational Design Patterns

  • Creational design patterns provide various object creation mechanisms, which increase flexibility and reuse of existing code.
  • They provide flexible ways to create objects while hiding the creation logic, thus promoting code reusability and maintainability.
Factory Method
 Concept
The Factory Method pattern defines an interface for creating an object, but allows subclasses to alter the type of objects that will be instantiated. It promotes loose coupling by eliminating the need to bind application-specific classes into the code.
Factory Method
 Problem Statement
An the initial logistics management application version only handles transportation by trucks, with most of the code residing within the Truck class. As the app gains popularity, requests to incorporate sea logistics arise, posing a challenge due to the tightly coupled nature of the code. Introducing a new transportation class becomes complex, requiring modifications throughout the codebase. This results in code cluttered with conditionals to accommodate different transportation types, leading to maintenance difficulties and decreased scalability.

Factory Design Pattern

Strengths Weaknesses
  • Encapsulates object creation.

  • Limits customization.
Use Cases
  • Centralized or encapsulated object creation.
  • Decoupling client code from concrete classes.
  • Supporting multiple object types.
Abstract Factory
 Concept
The Abstract Factory pattern is a creational design pattern that provides an interface for creating families of related or dependent objects without specifying their concrete classes. It allows you to encapsulate the creation of multiple related objects, ensuring that they are all of the same type and belong to the same family.

Abstract Factory Design Pattern

Strengths Weaknesses
  • Creates families of related objects.
  • Encapsulates object creation.
  • Promotes loose coupling.
  • Can be complex to implement.
  • May result in a large number of classes/interfaces.
Use Cases
  • Creating families of related objects.
  • Decoupling client code from concrete classes.
  • Supporting multiple object types and variations.

Factory Vs. Abstract Factory Design Pattern

  • In the Factory Method pattern, the TransportFactory creates instances of Truck or Ship based on the provided type.
  • In the Abstract Factory pattern, separate factories (TruckFactory and ShipFactory) are responsible for creatinginstances of their respective products (Truck and Ship).

Aspect Factory Method Abstract Factory

Usage Suitable when the creation logic may vary among subclasses. Suitable when the system needs to support multiple families of related objects.
Builder
 Concept
Builder is a creational design pattern that lets you construct complex objects step by step. The pattern allows you to produce different types and representations of an object using the same construction code.
Builder
 Problem Statement
For example, let’s think about how to create a House object. To build a simple house, you need to construct four walls and a floor, install a door, fit a pair of windows, and build a roof. But what if you want a bigger, brighter house, with a backyard and other goodies (like a heating system, plumbing, and electrical wiring)?

Builder Design Pattern

Strengths Weaknesses
  • Separates the construction of a complex object from its representation.
  • Allows the same construction process to create different representations.
  • Encapsulates the construction code, making it easy to change or extend.
  • Requires creating a concrete builder for each product variation, which can lead to class explosion.
  • May introduce complexity, especially for simple objects.
Use Cases
  • When the algorithm for creating a complex object should be independent of the object's parts and how they're assembled.
  • To construct objects step by step, allowing different configurations of the same object creation process.
  • When a class wants to avoid the telescoping constructor anti-pattern or the excessive number of constructors.

Structural Design Patterns

Structural Design Patterns

  • Structural design patterns focus on organizing objects and classes to form larger structures while keeping these structures flexible and efficient.
  • They address the composition of classes and objects to create new functionalities and relationships, promoting code reuse and maintainability.
Adapter
 Concept
The Adapter pattern allows incompatible interfaces to work together by wrapping the interface of a class with another interface. It acts as a bridge between two incompatible interfaces, converting the interface of a class into another interface expected by the client.
Adapter
 Problem Statement

Your stock market monitoring app currently downloads stock data from multiple sources in XML format.

To enhance the app's functionality, you decide to integrate a smart 3rd-party analytics library.

However, the analytics library only supports data in JSON format, presenting a compatibility challenge.

Adapter Design Pattern

Strengths Weaknesses
  • Converts interfaces to make objects work together.
  • Helps with integrating incompatible interfaces.
  • Enables collaboration between objects with different interfaces.
  • Can introduce additional complexity.
  • May require creating multiple adapters for different scenarios.
Use Cases
  • Integrating legacy systems with modern systems.
  • Working with third-party libraries with incompatible interfaces.
  • Adapting interfaces to meet specific system requirements.
Bridge
 Concept
The Bridge pattern separates the abstraction from its implementation, allowing them to vary independently. It enables you to change the implementation details of an abstraction without modifying its interface, promoting flexibility and decoupling.
Bridge
 Problem Statement

Bridge pattern splits a class hierarchy into two separate hierarchies—abstraction and implementation—allowing independent development.

For instance, extending a Shape class hierarchy with colors results in an exponential growth of subclasses, making maintenance difficult.

Bridge Design Pattern

Strengths Weaknesses
  • Decouples abstraction from implementation.
  • Allows for independent development of abstraction and implementation.
  • Facilitates easier maintenance and extensibility.
  • May increase complexity due to additional layers.
  • Requires careful design to define abstraction and implementation hierarchies.
Use Cases
  • When changes in the implementation should not affect the abstraction.
  • When a system should be extensible with minimal impact.
  • When abstraction and implementation hierarchies need to be independent.
Composite
 Concept
The Composite pattern composes objects into tree structures to represent part-whole hierarchies. It allows clients to treat individual objects and compositions of objects uniformly, simplifying the client code.
